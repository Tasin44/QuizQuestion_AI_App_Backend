PART 1 — backend/authapp/social_views.py (CORE GOOGLE LOGIN ENDPOINT)
====================================================================

1) Core imports needed for Google login
--------------------------------------

Code snippet:

from dj_rest_auth.registration.views import SocialLoginView
from allauth.socialaccount.providers.google.views import GoogleOAuth2Adapter
from allauth.socialaccount.providers.oauth2.client import OAuth2Client
from django.conf import settings
from rest_framework.permissions import AllowAny
from rest_framework.response import Response
from rest_framework import status
from allauth.socialaccount.models import SocialApp
from django.contrib.sites.models import Site
import logging

Explanation (easy):
- SocialLoginView: the ready-made REST API view (from dj-rest-auth) that performs the social login.
- GoogleOAuth2Adapter: tells django-allauth “use Google’s OAuth2 rules/endpoints”.
- OAuth2Client: helper used by allauth/adapters to do OAuth2 actions (token/code exchange).
- settings: used to read GOOGLE_CALLBACK_URL and DEBUG.
- AllowAny: users are not logged in yet, so this endpoint must allow unauthenticated requests.
- Response + status: lets your code return clean JSON error responses (instead of a generic server crash).
- SocialApp: database model where Google Client ID/Secret are stored (configured in Django Admin).
- Site: django-allauth uses Django “Sites” to pick the correct SocialApp.
- logging: logs unexpected failures.

Not required for Google login (only for Facebook):

from allauth.socialaccount.providers.facebook.views import FacebookOAuth2Adapter


2) CSRF exemption (API-friendly)
-------------------------------

Code snippet:

@method_decorator(csrf_exempt, name='dispatch')
class GoogleLogin(SocialLoginView):
    ...

Explanation:
- csrf_exempt disables CSRF checks for this view.
- This is common for API login endpoints called from a mobile app / SPA where CSRF cookies aren’t used.
- Google login can still work without it in some setups, but many API-only flows need this.


3) The GoogleLogin view (main required class)
--------------------------------------------

Code snippet:

class GoogleLogin(SocialLoginView):
    permission_classes = [AllowAny]
    authentication_classes = []

    adapter_class = GoogleOAuth2Adapter
    client_class = OAuth2Client

Explanation:
- class GoogleLogin(SocialLoginView):
  Uses dj-rest-auth’s SocialLoginView which contains the main social-login logic.

- permission_classes = [AllowAny]
  Anyone can call it (because user isn’t logged in yet).

- authentication_classes = []
  Disables DRF authentication for this endpoint so it doesn’t expect an existing token.

- adapter_class = GoogleOAuth2Adapter
  Selects Google as the provider.

- client_class = OAuth2Client
  Uses the OAuth2 client helper.

These lines are the core “must-have” lines for Google login.


4) Callback URL used for code-based flow
---------------------------------------

Code snippet:

def get_callback_url(self, request, app):
    return (
        getattr(settings, 'GOOGLE_CALLBACK_URL', None)
        or request.build_absolute_uri('/accounts/google/login/callback/')
    )

Explanation:
- This tells django-allauth what callback/redirect URL should be used when needed.
- If settings.GOOGLE_CALLBACK_URL is set, it uses it.
- Otherwise it builds a default callback URL based on the current request domain.

Note:
- callback_url matters most when you use an authorization “code” exchange flow.


5) The POST method: runs login + returns better errors
-----------------------------------------------------

Code snippet:

def post(self, request, *args, **kwargs):
    try:
        return super().post(request, *args, **kwargs)
    except Site.DoesNotExist:
        return Response(..., status=status.HTTP_500_INTERNAL_SERVER_ERROR)
    except SocialApp.DoesNotExist:
        return Response(..., status=status.HTTP_500_INTERNAL_SERVER_ERROR)
    except Exception as exc:
        logger.exception('Google social login failed')
        return Response(..., status=status.HTTP_400_BAD_REQUEST)

Explanation:
- return super().post(...)
  This is the IMPORTANT line: it triggers the real social login behavior implemented by SocialLoginView.

- except Site.DoesNotExist
  Means your Sites framework is enabled but the database has no Site row for your SITE_ID.

- except SocialApp.DoesNotExist
  Most common misconfiguration: you did not create/configure the Google SocialApp in Django Admin,
  or you didn’t attach it to the correct Site.

- except Exception
  Catches other errors (bad token, wrong callback URL, provider mismatch, etc.) and logs them.

Important:
- The exception blocks are not strictly required for login to work, but they make debugging much easier.


6) Optional redirect-based login view
------------------------------------

Code snippet:

from allauth.socialaccount.providers.oauth2.views import OAuth2LoginView

class GoogleLoginRedirectView(OAuth2LoginView):
    adapter_class = GoogleOAuth2Adapter
    client_class = OAuth2Client

    def get_callback_url(self, request, app):
        return (
            getattr(settings, 'GOOGLE_CALLBACK_URL', None)
            or request.build_absolute_uri('/accounts/google/login/callback/')
        )

Explanation:
- OAuth2LoginView is used for browser-style “redirect to Google” flows.
- If your frontend wants Django to start the redirect flow, this view helps.
- If your frontend/mobile already handles redirect and just POSTs token/code to GoogleLogin,
  then this view might be optional.


PART 2 — backend/aamyproject/settings.py (REQUIRED CONFIG)
=========================================================

1) Installed apps required for Google login
------------------------------------------

Code snippet (relevant lines):

INSTALLED_APPS = [
    ...
    'django.contrib.sites',
    'allauth',
    'allauth.account',
    'allauth.socialaccount',
    'allauth.socialaccount.providers.google',

    'dj_rest_auth',
    'dj_rest_auth.registration',
    ...
]

Explanation:
- django.contrib.sites: allauth uses the Sites framework to choose the correct SocialApp.
- allauth + account + socialaccount: core social-login system.
- allauth.socialaccount.providers.google: enables Google provider.
- dj_rest_auth + dj_rest_auth.registration: provides REST social login integration (SocialLoginView).


2) SITE_ID (very important)
--------------------------

Code snippet:

SITE_ID = int(os.getenv('SITE_ID', '3'))

Explanation:
- allauth selects the SocialApp (Google client id/secret) using the Site identified by SITE_ID.
- If SITE_ID points to a non-existing Site row, you get Site.DoesNotExist.


3) Authentication backends (required)
------------------------------------

Code snippet:

AUTHENTICATION_BACKENDS = (
    'django.contrib.auth.backends.ModelBackend',
    'allauth.account.auth_backends.AuthenticationBackend',
)

Explanation:
- Enables Django’s normal auth backend AND allauth backend.
- Without the allauth backend, social login won’t integrate properly.


4) Middleware requirement for allauth
------------------------------------

Code snippet:

MIDDLEWARE = [
    ...
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'allauth.account.middleware.AccountMiddleware',
    ...
]

Explanation:
- allauth requires AccountMiddleware.
- It must be placed after AuthenticationMiddleware.


5) Google provider configuration
-------------------------------

Code snippet:

SOCIALACCOUNT_PROVIDERS = {
    'google': {
        'SCOPE': ['profile', 'email'],
        'AUTH_PARAMS': {'access_type': 'online'},
        'OAUTH_PKCE_ENABLED': True,
    },
}

Explanation:
- SCOPE: asks Google for profile + email.
- AUTH_PARAMS: extra OAuth params.
- OAUTH_PKCE_ENABLED: enables PKCE (useful + more secure for SPA/mobile flows).

Not every key is mandatory, but this block is the normal place to configure Google login behavior.


6) JWT settings (not Google-specific but affects login response)
--------------------------------------------------------------

Code snippet (relevant lines):

REST_USE_JWT = True

REST_AUTH = {
    'USE_JWT': True,
    'JWT_AUTH_HTTPONLY': False,
    'JWT_AUTH_COOKIE': None,
    'JWT_AUTH_REFRESH_COOKIE': None,
}

SIMPLE_JWT = {
    ...
}

Explanation:
- These settings control how your API issues tokens (JWT access/refresh) after a successful login.
- Google login verifies the Google account; then your backend returns JWT tokens based on these settings.


7) SOCIALACCOUNT_LOGIN_ON_GET (optional)
---------------------------------------

Code snippet:

SOCIALACCOUNT_LOGIN_ON_GET = True

Explanation:
- For browser GET requests to allauth login URLs, it skips the “Continue” confirmation page.
- This is more relevant for redirect/browser flows than the API-only POST login.

IMPORTANT THINGS NOT SHOWN (BUT REQUIRED SOMEWHERE)
===================================================

1) You must have a URL route pointing to GoogleLogin
----------------------------------------------------
- Somewhere in your project urls.py you need something like:
  path('.../', GoogleLogin.as_view(), ...)
- Without it, the endpoint can’t be called.

2) You must configure Google SocialApp in Django Admin
------------------------------------------------------
- Django Admin -> Social applications -> Add
  - Provider: google
  - Client id: (from Google Cloud)
  - Secret: (from Google Cloud)
  - Sites: attach the Site matching SITE_ID

If this is missing you will hit your handled error:
- “Google SocialApp is not configured ...”
